# SkillGap.ai — Development Standards

## Critical Warnings

- **NEVER use Supabase MCP tools** (execute_sql, apply_migration, etc.) — the MCP is connected to the WRONG project (Inflow, not SkillGap). Always provide SQL in chat for the user to copy/paste.
- **YouTube embeds only.** Never download, re-host, or clip YouTube content. Use YouTube's official embed player. Timestamp cueing is fine. This is a legal hard line.
- **Never auto-publish rooms.** Auto-generated rooms go through an editorial review queue (`fp_room_blueprints` with status draft → approved → provisioned). Never skip the review step.

## What This Project Is

SkillGap.ai is where working professionals build AI fluency. Four systems working as one:

1. **Intelligence Engine** — Monitors AI landscape, detects trending topics, curates content from YouTube/Reddit/HN/RSS, generates learning paths before users search for them
2. **Learning Engine** — Task-first, function-and-fluency-adapted paths (30-60 min). Watch → Do → Check → Reflect. Missions use real tools, not simulations.
3. **Social Engine** — Live rooms where professionals sprint together. Three layers: always-on domain rooms (Heartbeat), scheduled tool sessions (Pulse), live events with tool providers (Spike).
4. **Growth Engine** — Shareable credentials, artifact attribution, room invite links, SEO-indexed path previews, weekly progress summaries, tool provider distribution, team workspaces.

For full product strategy, vision, and taxonomy: see @STRATEGY_Foundation.md

### Key Concepts

- **Functions** — Professional domains (Engineering, Marketing, Design, Product, Data, Sales, Operations). Users have one or more. The system adapts content per function.
- **Fluency levels** — Exploring → Practicing → Proficient → Advanced. Determines scaffolding depth, tool complexity, and mission independence.
- **Learning paths** — 3-5 module experiences generated by the curriculum generator (`lib/learn/curriculumGenerator.ts`). Three types: workflow-specific, general skill (function-adapted), trending/timely.
- **Rooms** — Persistent spaces (internally `fp_parties`) with AI hosts, sprint timers, and break content. Organized by professional domain, not by tool.
- **Synthetic users** — AI participants that create ambient activity. Fade as real density increases. Run every 2 min via cron.

## Code Conventions

### File Naming
- `lib/` files: `camelCase.ts` (e.g. `youtubeClient.ts`, `heatEngine.ts`)
- Route directories: `kebab-case` (e.g. `app/api/breaks/refresh-shelf/route.ts`)
- Components: `PascalCase.tsx` (e.g. `CreateRoomWizard.tsx`)
- Tests: `*.test.ts` co-located with source

### TypeScript
- Strict mode everywhere. Explicit parameter and return types on all exports.
- `interface` for object shapes, `type` for unions/intersections.
- No `any`. Use `unknown` + type guards.
- Named exports (not default), except React pages/components.
- JSDoc on all exported functions.

### React / Next.js
- App Router only. All routes in `app/`.
- Server Components by default. `'use client'` only when needed.
- API routes: `route.ts` with named `GET`, `POST`, `PATCH`, `DELETE` exports.
- Cron endpoints: GET handlers triggered by Vercel Cron.

### Design System
- Rules in `.claude/rules/ui-components.md` — **read before writing any UI.**
- Color source of truth: `lib/palette.ts` (JS) + `app/globals.css` (CSS custom properties).
- **NEVER** hardcode hex, Tailwind color defaults, or raw rgba.
- **ALWAYS** check `components/ui/` for existing components before writing inline JSX.
- Color philosophy: color = achievement or state. Everything else is grayscale.

## Database Conventions

- All tables prefixed `fp_`
- Column names: `snake_case`
- UUIDs for PKs (`gen_random_uuid()`)
- Timestamps: `TIMESTAMPTZ` with `DEFAULT now()`
- `JSONB` for flexible/nested data
- Admin client: `lib/supabase/admin.ts` (service role key)
- Always index columns used in WHERE clauses and JOINs

## AI/LLM Patterns

**Follow these exactly for any new AI integration.**

- **Model:** `gpt-4o-mini` for all completions. No exceptions without explicit approval.
- **Structured Outputs:** All LLM calls use `response_format: { type: 'json_schema' }` with strict schemas. Canonical example: `lib/breaks/scoring.ts`
- **Fail gracefully:** Every LLM call returns a safe default on error. Never crash the pipeline.
- **Safety:** All content evaluation prompts MUST include `SAFETY_PROMPT` from `lib/breaks/contentSafety.ts`. Hard requirement.
- **Cost controls:** Cap batch sizes (30 items max per eval run). Rate-limit per-room AI calls (480-720s cooldown for host messages). Cache aggressively — never re-evaluate already-scored content.
- **Creator respect:** Attribute all creators prominently. Comply immediately with removal requests.

## Cron Jobs

All background jobs: Vercel Cron → serverless GET handlers. Configured in `vercel.json`.

| Job | Route | Schedule |
|-----|-------|----------|
| Content Discovery | `/api/breaks/discover` | Every 6h |
| Content Evaluation | `/api/breaks/evaluate` | 2x daily (3am, 3pm UTC) |
| Shelf Refresh | `/api/breaks/refresh-shelf` | Daily 4am UTC |
| Topic Clustering | `/api/topics/cluster` | Hourly |
| Synthetic Tick | `/api/synthetics/tick` | Every 2 min |
| Background Rotate | `/api/backgrounds/rotate` | Daily 6am UTC |

Adding new crons: create `app/api/your-job/route.ts` → implement GET handler → optionally add POST for manual trigger → add to `vercel.json` → log start/end times.

## Guiding Principles

1. **Extend, don't rewrite.** Existing pipelines work. New systems layer on top. Don't refactor working code without a structural reason.
2. **Follow existing patterns.** Before writing a new module, read the equivalent existing one. `scoring.ts` for AI calls. `discovery.ts` for cron jobs. `shelf.ts` for DB operations. Match the style.
3. **Fail gracefully.** Every external call (LLM, YouTube API, Supabase) must have error handling that returns a safe default. Log errors, continue processing.
4. **Keep it simple.** No message queues, Redis, or worker infrastructure. Everything runs as Vercel serverless. If a function hits 60s, optimize it — don't add infrastructure.
5. **Canonical topics.** The topic taxonomy (`fp_topic_taxonomy`) is the backbone. Every piece of content, signal, and room should reference canonical topics. Freeform tags are being phased out.
6. **Cost awareness.** `gpt-4o-mini` for everything. Batch requests. Cache aggressively. Track token usage in logs.
7. **Ship incrementally.** Each system should be independently useful. Don't build monoliths.
