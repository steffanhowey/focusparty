Practice Loop — Service Blueprint

SkillGap.ai | Diagnosis of current state + redesigned experience | March 2026

1. Diagnosis: Why It Sucks

The practice loop has four structural failures. Every one of them breaks the core promise: "We help you actually DO things with AI tools."

Failure 1

Tool Challenges Are Theater

User clicks "Open Claude" → lands on claude.ai/new with no context, no pre-filled prompt, no instructions. Does whatever. Comes back. Clicks "I've completed this." Zero verification. The system literally cannot tell if they opened the tool or not.

Failure 2

Comprehension Is Decoration

One hardcoded question per content item. Always skippable. "Needs work" feedback still advances. No adaptive difficulty. No connection to what was just learned. It's a speed bump, not a checkpoint.

Failure 3

Paste-Back Is a Dead End

Textarea → paste code → get vague AI feedback → move on. No starter code. No scaffolding. No reference to which tool they should use. No incremental build. The AI evaluation gives a rating but it doesn't matter.

Failure 4

70% of Scaffolding Is Wasted

The LLM generates pre-watch challenges, key moments with timestamps, discussion questions, difficulty levels. None of it surfaces. The interleaver places exercises at arbitrary points with no correlation to content. All that intelligence sits in the DB unused.

4. The Three Practice Types — Redesigned

Each practice type has a distinct purpose and a distinct UI. No more one-size-fits-all PracticeViewer.

Quick Check

After every content item (inline, not a separate "step")
1-2 questions that validate understanding before moving on. Not skippable. Wrong answers trigger a hint + retry, not a block. Correct answers give a satisfying micro-animation and advance.
UI Behavior Appears as an overlay/card at the bottom of the content viewer — not a separate path step. Multiple choice OR short free-text. AI evaluates free-text. Wrong = hint text appears + try again. Right = checkmark animation + auto-advance after 1s. Max 2 attempts then reveal answer + advance.

Tool Mission

After a section of 2-3 content items (a dedicated path step)
The main event. User opens a real AI tool and completes a specific task. This is what makes SkillGap different — you actually use the tools. The prompt is pre-built, the criteria are clear, and proof of work is required.
UI Behavior Full-width mission briefing card. Prominent tool button with logo. Copy-to-clipboard on click. When user returns: paste output or upload screenshot. AI evaluates. Quality badge shown. Can iterate or advance. Skip available but tracked as "skipped" (no credit).

Reflection

End of a learning path section (before section milestone)
Open-ended prompt that connects what was learned to the user's actual work/goals. "How would you apply [concept] to [their stated goal]?" AI gives thoughtful feedback, not a grade.
UI Behavior Clean textarea with the reflection prompt. Min 50 characters to submit. AI responds conversationally (not as an evaluator). Response shown inline. Always advances after submission. Optional: save as a "learning note" to their profile.

5. Tool Mission — Detailed Flow Specification

This is the centerpiece of practice. Every detail matters.

Element Specification Why
Transition In After last content item in section, show transition card: tool logo, "Time to practice" heading, estimated time (from scaffolding), "Start Mission" button Context switch needs a beat. User should know what's coming and mentally shift from passive to active mode.
Mission Briefing Objective: 1 sentence, action-oriented ("Build a prompt that...")
Context: 1-2 sentences connecting to content just watched
Pre-built prompt: Shown in a styled code block. Copy button prominent. The prompt is specific to the content — not generic.
Instructions: 3-5 numbered steps
Success criteria: Checkable list of what "done" looks like
Tool CTA: Large button with tool logo: "Open in Claude" / "Open in Cursor" / etc. 		The user should never wonder "what am I supposed to do?" Every piece of info they need is on screen before they leave.
Prompt Design The pre-built prompt is NOT just the exercise instructions. It's a ready-to-paste prompt for the target tool. Example for Claude:

"I just learned about [topic from video]. I want to practice [specific skill]. Help me [specific task]. Here's my starting point: [starter code/context]. Guide me step by step."

Generated by a prompt builder that takes: scaffolding exercise data + content metadata + user's role/goal from profile. 		The #1 barrier to practice is "I don't know how to start." A pre-built prompt eliminates that barrier completely. This is the difference between homework and guided practice.
Tool Launch Click tool CTA → navigator.clipboard.writeText(prompt) → open tool URL in new tab → show toast: "Prompt copied! Paste it in Claude."

Tool URLs from registry. Where deep-linking is supported (v0 has prompt params), use it. 		Copying + opening in one click reduces friction from 4 steps to 1. Toast confirms the copy worked.
While Practicing SkillGap tab shows a persistent reference bar:
• Mission objective (pinned)
• Success criteria checklist
• Elapsed timer
• "Re-copy prompt" button
• "I'm done" button 		User will alt-tab back to reference instructions. This must be immediately visible — not buried under content they already watched.
Submission User clicks "I'm done" → submission panel:
Option A: Paste text output (large textarea)
Option B: Upload screenshot (drag-drop zone)
Both: "Submit for review" button
Skip: Available as text link ("Skip this practice") — tracked as skipped, no credit, no achievement XP. 		Proof of work is what makes practice real. But it must be frictionless — paste or screenshot, nothing more complex.
AI Evaluation Text submissions: GPT-4o-mini evaluates against success criteria. Returns structured response: quality (nailed_it / good / needs_iteration), per-criterion pass/fail, specific feedback, suggestion for improvement.
Image submissions: GPT-4o-mini with vision analyzes screenshot against criteria. Same structured output.
Schema: Structured Outputs with strict JSON schema. 		Feedback must be specific ("Your prompt structure is strong but missing error handling criteria") not vague ("Good job!"). This is where the AI proves its value.
Feedback Display Quality badge with color: 🟢 Nailed it / 🟡 Good / 🟠 Needs iteration
Per-criterion breakdown (checkmarks / x marks)
Specific feedback paragraph
Two CTAs: "Try Again" (goes back to mission briefing) or "Continue" (advances)
"Nailed it" gets a celebration micro-animation. 		Users need to see exactly where they hit and missed. "Try Again" creates a real practice loop — not just "mark complete and move on."
Key principle: Practice should never block progression. "Skip" is always available. But skipping means no credit, no badge, no XP. The incentive is intrinsic (learn the skill) + extrinsic (achievements, profile, shareable badges). Never punitive.

6. What Changes in the Codebase

Mapping blueprint requirements to specific technical changes.

AreaCurrent StateRequired Change
PracticeViewer.tsx Single component handles all 3 types. Tool challenges just show buttons. "Mark Complete" with no input. Delete and rebuild as 3 components: QuickCheck (inline overlay), ToolMission (full mission briefing), Reflection (clean textarea). Each with purpose-built UI.
toolRegistry.ts 8 tools with static homepage URLs. No prompt templates. No deep-link patterns. Add per-tool: prompt template function, deep-link URL builder (where supported), icon asset path, clipboard format. Tool URLs should be contextual, not static.
practiceInterleaver.ts Hardcoded: comprehension after every 2nd item, exercises at section boundaries. No correlation to content. Redesign: Quick Checks after every content item (inline, not separate steps). Tool Missions at section ends (only if scaffolding has exercises). Reflections before section milestones. Use scaffolding difficulty to sequence.
Prompt Builder Doesn't exist. Exercises are shown raw from scaffolding. New module: lib/learn/promptBuilder.ts — takes scaffolding exercise + content metadata + user profile → generates a ready-to-paste tool prompt specific to the content.
/api/learn/evaluate Text-only evaluation. No support for tool challenges. No image/vision support. Extend: Accept image uploads (base64). Use GPT-4o-mini with vision for screenshot evaluation. Return per-criterion breakdown, not just a quality rating.
fp_learning_progress Tracks started_at and completed. No submission data. Add columns: submission_text, submission_image_url, evaluation_result (JSONB), quality_score, time_spent_seconds, skipped (boolean).
Path page (paths/[id]) Renders PracticeViewer for practice items. No transition cards. No persistent reference bar. Add: Transition card between content → practice. Persistent reference bar during tool missions. Submission panel. Feedback card with iteration support.
Quick Checks Separate path step. Skippable. One question. Move inline: Overlay at bottom of content viewer. Not a path step. Not skippable (but max 2 attempts then reveal). Can be multiple choice or free-text. Pull from scaffolding comprehension_checks array (use ALL of them, not just first).

7. Success Metrics

How we know the redesign is working.

MetricCurrentTargetWhy It Matters
Practice completion rate Unknown (no real tracking — "Mark Complete" is meaningless) >40% of users who encounter a Tool Mission submit work If users actually do the practice, the product is working
Practice skip rate Unknown (skip and complete are indistinguishable) <50% skip rate on Tool Missions If more than half skip, the missions aren't compelling enough
"Nailed it" rate N/A 25-35% on first attempt Too high = too easy. Too low = unclear instructions. Sweet spot is challenging but achievable.
Iteration rate 0% (no iteration possible) >15% of non-"nailed it" users retry Retry means the user cares about getting it right. This is real learning.
Tool click-through rate Unknown >60% of users who see a Tool Mission click the tool CTA The mission briefing must be compelling enough to get them to actually open the tool
Quick Check pass rate Unknown 70-80% on first attempt Quick Checks validate content understanding. Below 70% = content or question quality issue.

8. Implementation Priority

What to build first, and why.

Sprint 1 — Fix Tool Missions

The Big Win

Mission briefing UI + clipboard copy + submission panel + AI evaluation with criteria. This single sprint transforms practice from theater to real. Most user-visible impact.

Sprint 2 — Fix Quick Checks

Make Content Stick

Move checks inline (not separate path steps). Make them non-skippable with 2-attempt limit. Use ALL comprehension checks from scaffolding. Add hint system.

Sprint 3 — Prompt Builder

Intelligence Layer

Context-aware prompt generation that incorporates content metadata + user profile + scaffolding data. Makes every Tool Mission feel custom-built for the user.

Sprint 4 — Reflections + Polish

Depth

Add end-of-section reflections. Connect to user goals. Save as learning notes. Polish transitions and micro-animations. Wire into achievement system.
