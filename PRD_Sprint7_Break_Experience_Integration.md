# Sprint 7: Break Experience Integration

**Goal:** Surface the content intelligence engine's output — scaffolding, curriculum sequencing, host scripts, and topic-aware content — in the user-facing break experience. Everything built in Sprints 1–6 becomes visible to learners for the first time.

**Scope:** Frontend integration + minimal backend wiring. No new AI calls, no new cron jobs, no new tables. The data already exists in the database. This sprint connects it to the UI.

---

## Context: What's Built vs. What Users See

The content pipeline produces rich learning data for every piece of break content:

- **Scaffolding** — pre-watch challenges, key moments with timestamps, comprehension checks, practice exercises, discussion questions (stored in `fp_break_content_items.scaffolding` JSONB, copied from `fp_break_content_scores` during shelf promotion in `lib/breaks/shelf.ts` line 255)
- **Curriculum sequences** — ordered learning progressions with rationale per video (stored in `fp_room_blueprints.curriculum_sequence` JSONB, generated by `lib/rooms/contentSelector.ts`)
- **Host scripts** — topic-specific host personalities with per-trigger messages (stored in `fp_room_blueprints.host_config` JSONB, generated by `lib/rooms/hostScriptGenerator.ts`)
- **Topic intelligence** — canonical topics, heat scores, trending status (stored in `fp_topic_clusters`, surfaced via `fp_parties.is_trending` and `fp_parties.topic_slug`)

**None of this reaches the user.** The break experience today is: pick a category → pick a duration → pick a video card → watch a YouTube embed in a draggable overlay → return to sprint. The `BreakContentItem` type in `lib/types.ts` (line 479) has no `scaffolding` field. The `BreakVideoOverlay` component renders only the video player with no learning context. Auto-generated rooms with curricula serve content the same way as manually created rooms — random shelf pulls by world key.

This sprint closes that gap across three layers.

## Deliverable 2: Pre-Watch Challenge Card

**Why:** The scaffolding's `preWatch` field contains a concrete challenge the learner should attempt before watching. This is the highest-leverage scaffolding component — it activates prior knowledge and creates productive struggle in under 5 minutes. It shows before the video starts playing.

**File:** New component `components/environment/PreWatchCard.tsx`

**What it renders:**

A dismissible card that appears between content selection and video playback. It's NOT a gate — the learner can skip it immediately. The card contains:

- Header: "Before you watch" (small text, `text-white/50`)
- The `preWatch.prompt` text (the actual challenge — 1-3 sentences)
- Context line: `preWatch.context` (italic, `text-white/30`, explains why this challenge matters)
- Time estimate: `preWatch.estimatedMinutes` min (pill badge)
- Two actions: "I'll try it" (starts a visible timer, doesn't block video) and "Skip" (dismisses card, video plays immediately)

**Behavior:**

- Appears as an overlay on the `BreakVideoOverlay` component area, before the YouTube player loads
- When "I'll try it" is pressed: card stays visible with a countdown timer, video does NOT auto-play yet. After the timer completes (or user clicks "Ready to watch"), video begins
- When "Skip" is pressed: card dismisses, video auto-plays immediately (current behavior)
- If scaffolding is null or `preWatch` is missing: component renders nothing (returns `null`). This is the default for legacy content.

**Props:**

```typescript
interface PreWatchCardProps {
  preWatch: PreWatchChallenge;
  onReady: () => void;    // Called when user is ready to watch (skip or timer complete)
  onSkip: () => void;     // Called when user explicitly skips
}
```

**Design rules:**
- Use CSS variables for all colors (`var(--color-*)`)
- Glass surface: `background: rgba(10,10,10,0.85)`, `backdropFilter: blur(24px)` (matches existing `BreakVideoOverlay` style)
- Use `shadow-float` token for box shadow
- No raw hex values. No Tailwind color defaults.
- Import `Button` from `@/components/ui/Button` for the action buttons (variant: `ghost` for skip, `secondary` for "I'll try it")

## Deliverable 4: Post-Watch Learning Panel

**Why:** After the video plays (or when the user is ready to end the break), show comprehension checks and the practice exercise. This transforms a passive video break into an active learning moment. The panel is optional — the user can dismiss it and return to sprint at any time.

**File:** New component `components/environment/PostWatchPanel.tsx`

**What it renders:**

A slide-up panel that appears when the break video finishes OR when the user clicks "End Break" (whichever comes first). It replaces the current immediate transition to "resuming" phase. The panel has three sections, shown sequentially (not all at once):

**Section 1: Comprehension Check** (shown first)
- Header: "Quick check"
- Shows one question from `scaffolding.comprehensionChecks[]` (cycle through them — first check on first break, second on next, etc.)
- The question text
- A "Show answer" toggle that reveals `check.answer`
- If `check.timestampHint` exists, show a "Rewatch this part" link that seeks the video to that timestamp
- "Next" button advances to Section 2

**Section 2: Practice Exercise** (shown after comprehension check)
- Header: "Try it yourself" with time estimate pill (`exercise.estimatedMinutes` min)
- `exercise.prompt` (the main challenge)
- `exercise.instructions[]` rendered as a numbered list
- `exercise.tools[]` shown as small pill badges
- If `exercise.starterCode` exists, show in a monospace code block
- `exercise.successCriteria` as a subtle footer line
- "Done" and "Skip" buttons

**Section 3: Discussion Question** (shown after exercise or skip)
- Header: "For the room"
- `scaffolding.discussionQuestion` text
- "Share in chat" button → posts the discussion question to the room's activity feed as a system event (type: `discussion_prompt`, actor: `system`)
- "Back to work" button → transitions to "resuming" phase (existing 3-second countdown)

**Behavior:**
- Each section animates in (simple opacity + translateY transition)
- User can skip any section
- The "Back to work" button is always visible at the bottom as a secondary action
- If scaffolding is null: skip the entire panel, go directly to "resuming" phase (preserves current behavior exactly)
- Panel auto-dismisses after 60 seconds if no interaction (prevents blocking the sprint timer indefinitely)

**Props:**

### 6a: Curriculum Data Hook — `lib/useCurriculum.ts`

**What it does:** Fetches the curriculum sequence for a room from its blueprint, and tracks the user's progress through it.

```typescript
export interface CurriculumEntry {
  videoId: string;
  title: string;
  creator: string;
  sequencePosition: number;
  learningRationale: string;
  durationSeconds: number;
  tasteScore: number;
  scaffolding: Scaffolding | null;
  completed: boolean;          // Has this user watched this video?
}

export interface UseCurriculumReturn {
  curriculum: CurriculumEntry[] | null;  // null = no curriculum (manual room)
  currentPosition: number;               // Next unwatched position
  loading: boolean;
  markCompleted: (videoId: string) => void;
}

export function useCurriculum(
  partyId: string | null,
  userId: string | null
): UseCurriculumReturn
```

**How it loads the curriculum:**

1. Fetch the party's `blueprint_id` from `fp_parties`
2. If no `blueprint_id`: return `{ curriculum: null }` — this is a manual room, no curriculum
3. Fetch `curriculum_sequence` and `content_selection` from `fp_room_blueprints` where `id = blueprint_id`
4. Cross-reference with `fp_break_content_items` to get current shelf items matching the curriculum video IDs (the shelf may have newer content too — curriculum entries that are also on shelf get scaffolding data from the shelf item)
5. Check `fp_break_engagement` for this user to determine which videos they've already completed (event_type = 'completed' for videos in this curriculum)
6. Return the ordered curriculum with completion state

**API route:** New route `app/api/breaks/curriculum/route.ts`

```
GET /api/breaks/curriculum?party_id=...
```

Returns `{ curriculum: CurriculumEntry[] | null }`. Requires authentication (uses `createClient` from `lib/supabase/server`).

**How it resolves curriculum entries to shelf items:**

The blueprint's `curriculum_sequence.items` stores `videoId` (YouTube external ID), `title`, `creator`, `sequencePosition`, `learningRationale`. The shelf items (`fp_break_content_items`) store the full `video_url` which contains the YouTube ID. Match by extracting the YouTube ID from `video_url` (use existing `extractYouTubeId` from `lib/youtube.ts`).

If a curriculum video is not currently on the shelf (expired or not yet promoted), still show it in the curriculum list but mark it as unavailable. The user sees the progression but can only play items that are on the active shelf. This prevents gaps in the learning sequence.

**Completion tracking:**

When a user completes a break with a curriculum video (engagement event_type = 'completed'), the curriculum hook marks that position as done. Completion state is derived from existing `fp_break_engagement` rows — no new table needed. Filter by: `user_id`, `content_item_id` matching a shelf item whose `video_url` contains the curriculum's `videoId`, `event_type = 'completed'`.

Store `currentPosition` in component state (first uncompleted position). This resets on page reload by recalculating from engagement data.

## Deliverable 7: Host Break Intros for Auto-Generated Rooms

**Why:** Auto-generated room blueprints include host scripts with per-trigger messages (`triggerHints`). The `session_started` hint includes topic-specific welcome context, and `sprint_near_end` can preview the next break content. Today, the host system uses `lib/hostPrompt.ts` which calls GPT-4o-mini to generate messages. For auto-generated rooms, the blueprint already contains pre-written guidance — the LLM should use it.

**File:** Modify `lib/hostPrompt.ts`

**Changes:**

1. When generating a host prompt for a room that has a blueprint, inject the blueprint's `triggerHints` into the system prompt as additional context. The host LLM doesn't use the hints verbatim — they're guidance that shapes the generated message.

2. The `HostGenerationInput` (or whatever the input type is called in `hostPrompt.ts`) already receives room context. Add an optional field:

```typescript
/** Pre-written host guidance from auto-generated blueprint (injected into LLM context) */
blueprintHints?: {
  triggerHint: string;        // The specific hint for this trigger type
  curriculumContext?: string; // e.g. "Next break: 'Multi-File Editing with Cursor' by Fireship"
};
```

3. In the system prompt construction, if `blueprintHints` is present, append:

```
ROOM-SPECIFIC GUIDANCE (from the room's learning design):
${blueprintHints.triggerHint}

${blueprintHints.curriculumContext ? `UPCOMING CONTENT:\n${blueprintHints.curriculumContext}` : ''}
```

4. When the host is triggered at `sprint_near_end` in a room with a curriculum, include the next video's title and creator in `curriculumContext` so the host can preview the upcoming break content.

**Where to resolve blueprint hints:**

The caller of `generateHostPrompt` (likely in the environment page or a host trigger hook) needs to:
1. Check if the room's `blueprint_id` exists (from party data)
2. If yes, load the blueprint's `host_config.triggerHints` (from `roomDna` cache — `getDynamicHostConfig(worldKey)`)
3. Map the current trigger type to the corresponding hint
4. Pass it as `blueprintHints` to the prompt generator

The `HostConfig` type (from `lib/hosts.ts`) already has `styleGuide.triggerHints` which is populated by `lib/roomDna.ts` during blueprint parsing (see `roomDna.ts` line 216). So the data is already in the runtime cache. The caller just needs to read it and pass it through.

**Error handling:** If hints are missing or the blueprint has no triggerHints for this trigger type, omit the section entirely. The LLM generates a generic message as it does today. No degradation.

## Deliverable 8: Scaffolding Engagement Tracking

**Why:** Sprint 6 defined `fp_scaffolding_events` for tracking scaffolding interactions. This deliverable wires the frontend to emit those events so the closed-loop calibration can measure scaffolding effectiveness.

### 8a: Tracking Utility — `lib/scaffoldingTracking.ts`

```typescript
type ScaffoldingEventType =
  | "pre_watch_shown"
  | "pre_watch_attempted"    // User clicked "I'll try it"
  | "pre_watch_skipped"      // User clicked "Skip"
  | "key_moment_clicked"     // User clicked a moment marker
  | "comprehension_shown"
  | "comprehension_answer_revealed"
  | "comprehension_rewatch"  // User clicked "Rewatch this part"
  | "exercise_shown"
  | "exercise_completed"     // User clicked "Done"
  | "exercise_skipped"
  | "discussion_shared"      // User shared question to chat
  | "post_watch_dismissed"   // User dismissed entire panel
  | "post_watch_auto_dismissed"; // 60s timeout

export function trackScaffoldingEvent(
  contentItemId: string,
  eventType: ScaffoldingEventType,
  metadata?: Record<string, unknown>
): void {
  // Fire-and-forget POST to /api/breaks/scaffolding-events
  fetch("/api/breaks/scaffolding-events", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      content_item_id: contentItemId,
      event_type: eventType,
      metadata: metadata ?? null,
    }),
  }).catch(() => {}); // Swallow errors — never block UI
}
```

### 8b: API Route — `app/api/breaks/scaffolding-events/route.ts`

```
POST /api/breaks/scaffolding-events
Body: { content_item_id, event_type, metadata? }
```

**Logic:**
1. Authenticate user (same pattern as `app/api/breaks/engagement/route.ts`)
2. Validate `event_type` against allowed enum values
3. Insert into `fp_scaffolding_events` table (created in Sprint 6): `{ content_item_id, user_id, event_type, metadata, created_at }`
4. Return `{ ok: true }`

If `fp_scaffolding_events` table doesn't exist yet (Sprint 6 may have defined it differently), create it:

```sql
CREATE TABLE IF NOT EXISTS fp_scaffolding_events (

CREATE INDEX idx_scaffolding_events_content ON fp_scaffolding_events(content_item_id);
CREATE INDEX idx_scaffolding_events_user ON fp_scaffolding_events(user_id);
CREATE INDEX idx_scaffolding_events_type ON fp_scaffolding_events(event_type);
```

Provide this SQL in chat for the user to run — do NOT use Supabase MCP tools.

## Deliverable 9: Scaffolding-Aware Taste Profile Updates

**Why:** The taste profile system (`lib/breaks/tasteProfile.ts`) currently adjusts per-topic weights based on video completion/abandonment. Scaffolding engagement should amplify the signal — a user who completes the exercise is more engaged with that topic than one who only watched the video.

**File:** Modify `lib/breaks/tasteProfile.ts`

**Changes to `updateTasteProfile` function:**

After processing the standard engagement events (completed/abandoned/extended), check for scaffolding events for the same content items and apply bonus deltas:

| Event | Delta (per topic on the content item) |
|-------|---------------------------------------|
| `exercise_completed` | +0.10 bonus (on top of video completion) |
| `pre_watch_attempted` | +0.05 bonus |
| `comprehension_rewatch` | +0.05 bonus |
| `discussion_shared` | +0.05 bonus |
| `exercise_skipped` + `post_watch_auto_dismissed` | -0.05 penalty |

**Implementation:**

1. After the existing engagement event processing loop, query `fp_scaffolding_events` for events since the user's last profile update, joining on `content_item_id` to get topics from `fp_break_content_items`
2. Accumulate additional deltas per topic
3. Merge with the existing delta map before upserting to `fp_user_taste_profiles`
4. Clamp final weights to `[-1.0, 1.0]` (existing clamping logic)

**Error handling:** If the scaffolding events query fails, log and continue. The base taste profile update from video engagement still works. Scaffolding bonus is additive, never required.

## Deliverable 10: Presence Broadcasting for Break Learning State

**Why:** Other participants in the room can see when someone is on a break (the presence system already broadcasts `breakContentTitle`). Extend this to show what scaffolding activity they're doing — "doing pre-watch challenge", "trying the exercise" — for social accountability and curiosity.

**File:** Modify `lib/presence.ts`

**Changes:**

1. Add an optional `breakLearningState` field to the presence payload:

```typescript
breakLearningState?: "pre_watch" | "watching" | "comprehension" | "exercise" | "discussion" | null;
```

2. In the presence payload builder (where `phase === "break"` is already handled), set `breakLearningState` based on the current scaffolding UI state from the environment page.

3. In `lib/presence.ts`'s display text function (the one that generates human-readable status strings), add cases:

```
pre_watch → "preparing to learn"
watching → "watching" (existing)
comprehension → "checking understanding"
exercise → "trying the exercise"
discussion → "reflecting"
```

4. In the environment page, update the presence payload when scaffolding state changes:

```typescript
// When pre-watch is shown
breakLearningState: showPreWatch ? "pre_watch" :
                    showPostWatch ? (currentSection === "comprehension" ? "comprehension" :
                                    currentSection === "exercise" ? "exercise" :
                                    "discussion") :
                    breakActive ? "watching" : null
```

**Backwards compatibility:** The `breakLearningState` field is optional. Older clients that don't send it will show the existing "on break" status. The presence channel handles unknown fields gracefully (Supabase Realtime passes through arbitrary JSON).

## Notes for the AI Engineer

1. **The data is already there.** The API route (`app/api/breaks/content/route.ts`) does `select("*")` from `fp_break_content_items`. The `scaffolding` JSONB column is already returned in every response. The shelf promotion code (`lib/breaks/shelf.ts` line 255) already copies scaffolding from scores to shelf items. You're wiring up the frontend, not building a pipeline.

2. **Null-check everything.** The `scaffolding` field will be null for: legacy seed content, content that failed scaffolding generation, content promoted before Sprint 3, and the "reset", "reflect", "move" categories. Every component must render cleanly when scaffolding is null. The entire integration is additive — zero visual change for unscaffolded content.

3. **Don't modify the break timer behavior.** The sprint timer pauses when break starts and resumes when the user returns. The post-watch panel is part of the break phase. Don't un-pause the sprint timer during post-watch. Don't add a separate timer. The user controls when they return.

4. **The curriculum hook needs careful video ID matching.** The blueprint stores `externalId` (YouTube video ID). The shelf items store `video_url` (full YouTube URL). Use `extractYouTubeId()` from `lib/youtube.ts` to match them. Account for both `youtube.com/watch?v=` and `youtu.be/` formats — the existing utility handles both.

5. **roomDna already caches blueprint data.** Don't re-fetch blueprints. Use `getDynamicHostConfig(worldKey)` from `lib/roomDna.ts` for host trigger hints. The cache is populated during provisioning (`setRoomDna` in `lib/roomFactory/provision.ts` line 268). For the curriculum, you do need a fresh fetch (curriculum data is in `curriculum_sequence` which isn't part of the roomDna cache), but this is a one-time load per room session.

6. **Follow the UI component rules.** `Button` from `@/components/ui/Button`, `Card` from `@/components/ui/Card`. CSS variables for colors. Shadow tokens. No raw hex. No Tailwind color defaults. Read `.claude/rules/ui-components.md`.

7. **The `discussion_prompt` activity event type may not exist yet.** Check `lib/types.ts` or wherever activity event types are defined. If it doesn't exist, add it to the union type. The activity feed components will need to handle rendering this new event type (show the discussion question text in the feed).

8. **Don't touch cron jobs.** This sprint has no cron changes. No vercel.json changes. The scaffolding events table is the only database change, and provide the SQL in chat for the user to run manually.

9. **Engagement tracking is fire-and-forget.** Both the existing `trackEngagement()` function in `BreakVideoOverlay.tsx` and the new `trackScaffoldingEvent()` use `.catch(() => {})`. Never let a tracking failure block the UI. Never await these calls in the render path.

10. **Test with existing scaffolded content.** Before building the UI, verify that scaffolded shelf items exist in the database. Query: `SELECT id, title, scaffolding_status FROM fp_break_content_items WHERE scaffolding IS NOT NULL LIMIT 5;`. If zero results, the pipeline hasn't promoted any scaffolded content to the shelf yet — run the shelf refresh cron or manually insert a test item.
